# P3.1 — Asset file size in the media panel

**Seed:** `tests/e2e/seed.spec.ts`
**Roadmap status:** ✅ done (`76d78db`)

## Application Overview

The left-side media panel lists every asset in the project. Each row shows the asset's label, duration, **and file size** in human-readable form (e.g. `12.4 MB`). The size comes from the `sizeBytes` field on the `AxcutAsset` schema, populated at import time via `fs.stat`.

## Test Scenarios

### 1. Asset with a real file shows its size in the panel

**Setup**

1. Open the editor at `http://localhost:5173/?windowType=editor`.
2. Seed the browser shim with a project that contains a video asset pointing at a real file on disk (e.g. `tests/fixtures/sample.webm`).

**Steps**

1. Locate the asset row in the left media panel.
2. Read the visible size string.

**Expected results**

- The size string is **not** the em-dash placeholder (`—`).
- The size string matches a `B / KB / MB` formatted number consistent with the file's actual size (sample.webm is ~300 KB, so expect `~300 KB`).

### 2. Asset with no size (legacy / missing) shows the em-dash

**Setup**

1. Same as scenario 1, but with a project whose asset has no `sizeBytes` field.

**Steps**

1. Locate the asset row in the panel.

**Expected results**

- The size string is the em-dash `—`.
- No console error from the panel.

### 3. Round-trip — adding an asset populates `sizeBytes`

**Setup**

1. Open the editor.
2. Trigger `addAsset` via the IPC bridge with a real file path.

**Steps**

1. Call `nativeBridge.invoke({ domain: "ai-edition", action: "addAsset", payload: { path: "<abs path>", label: "Test" } })`.
2. After the call resolves, read the document's `assets` array via the project store (exposed in browser-shim mode as `window.__OPENSCREEN__?.getDocument?.()` — or directly by reading the `localStorage["browser-shim-document"]` key).

**Expected results**

- The newly added asset has a `sizeBytes` field with a positive integer value.
- The value matches `fs.stat(path).size` (within ±0).
